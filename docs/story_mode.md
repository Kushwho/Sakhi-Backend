# Sakhi Story Mode: Implementation Guide

This document describes the Story Narration Mode — how it is architected, how stories are stored and fetched, and how the frontend should integrate with it.

---

## Overview

Story Mode lets a child listen to a pre-loaded story narrated by a Sakhi voice agent. The user selects a story genre, previews the title, confirms it, and then joins a LiveKit voice room where the agent narrates the story segment-by-segment — with pauses for interaction in between.

**Key design decisions:**
- Stories are stored in PostgreSQL (seeded by admins), not generated by AI.
- All story segments are pre-fetched from DB into agent memory **before** the voice session starts — zero DB calls during narration.
- The Story Agent runs on its own `AgentServer` (separate process from the voice agent).

---

## User Flow

```
1. Child picks a genre (adventure, fable, animal, etc.)
        ↓
2. Frontend: GET /api/stories/random?genre=adventure
   → Returns a random story: { id, title, genre, total_segments }
        ↓
3. Frontend shows the title: "How about 'The Clever Fox'?"
   Child: likes it → confirm | doesn't like it → call again for another
        ↓
4. Frontend: POST /api/story-token  { "story_id": "uuid" }
   → Returns: { token, room_name, livekit_url }
        ↓
5. Frontend connects to LiveKit room (same as voice mode)
        ↓
6. story-agent connects, pre-fetches ALL segments from DB into memory
        ↓
7. Agent greets child, narrates Part 1, interacts, narrates Part 2...
        ↓
8. Story ends → agent congratulates child → session closes → summary saved to DB
```

## Database Schema

Two tables are added to the existing schema in `db/migrations.py`:

### `stories`

| Column | Type | Description |
|--------|------|-------------|
| `id` | UUID | Primary key |
| `title` | TEXT | Story title shown to the child |
| `genre` | TEXT | e.g. `fable`, `adventure`, `animal`, `fantasy`, `mythology`, `moral` |
| `age_min` | INT | Minimum recommended age |
| `age_max` | INT | Maximum recommended age |
| `language` | TEXT | `English`, `Hindi`, etc. |
| `total_segments` | INT | Total number of segments (auto-set on insert) |
| `created_at` | TIMESTAMPTZ | Creation timestamp |

### `story_segments`

| Column | Type | Description |
|--------|------|-------------|
| `id` | UUID | Primary key |
| `story_id` | UUID | FK → `stories.id` (cascade delete) |
| `position` | INT | 1-indexed order of the segment |
| `content` | TEXT | The actual narration text for this segment |

A **unique constraint** on `(story_id, position)` prevents duplicate segments.

## Seeding Stories

Use the `story_emitter.py` script at the project root to seed stories directly into the database:

```bash
python story_emitter.py
```

Stories inserted by the seeder will automatically appear in all browse/filter API responses.

> **Tip:** Keep each segment to 2-4 sentences — short enough for the child to listen to comfortably before the agent pauses for a question.

## API Endpoints

All story endpoints are registered in `api/story_routes.py` and mounted in `api/routes.py`.

### `GET /api/stories/random`

Returns a random story for preview. The frontend shows the title and asks the child if they want to hear it. If not, call again to get a different random pick.

**Query parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `genre` | string | `""` | Filter by genre |
| `age` | int | `0` | Filter stories where `age_min ≤ age ≤ age_max` |
| `language` | string | `"English"` | Filter by language |

**Response (200 OK):**

```json
{
  "id": "baa21f56-e4c5-4ba9-9d5a-13cb86668b9f",
  "title": "The Clever Little Fox",
  "genre": "fable",
  "age_min": 4,
  "age_max": 8,
  "total_segments": 4
}
```

**404** — if no stories match the filters.

---

### `GET /api/stories`

Lists all available stories. Accepts the same optional query filters (`genre`, `age`, `language`).

**Response (200 OK):**

```json
{
  "stories": [ { "id": "...", "title": "...", "genre": "...", ... } ],
  "total": 3
}
```

---

### `GET /api/stories/{story_id}`

Returns metadata for a single story. Use this to confirm story details before starting a session.

**Response (200 OK):** Same shape as the random story response.  
**404** — if story not found.

---

### `POST /api/story-token`

Generates a LiveKit room token and dispatches the `story-agent`. **Requires child profile auth.**

**Request Headers:**
```
Authorization: Bearer <profile_token>   (child profile type required)
```

**Request Body (JSON):**

```json
{ "story_id": "baa21f56-e4c5-4ba9-9d5a-13cb86668b9f" }
```

**Response (200 OK):**

```json
{
  "token": "eyJhbGciOiJIUzI1NiIsIn...",
  "room_name": "story-storykid-1741810877",
  "livekit_url": "wss://your-project.livekit.cloud"
}
```

**Errors:**
- `403` — token belongs to a parent profile (only child profiles allowed)
- `404` — `story_id` does not exist in the database
- `500` — LiveKit credentials not configured

*Frontend action:* Pass `token` and `livekit_url` into the LiveKit room component — identical to voice mode.

## How the Story Agent Works

### At Session Start (Before Voice Pipeline)

1. Agent connects to the LiveKit room.
2. Reads `story_id` and child profile data from **room metadata**.
3. Calls `get_story()` + `get_all_segments()` — **one DB round-trip**.
4. All N segments are stored in `agent._segments` (an in-memory list).
5. Voice pipeline starts.

### During Narration (Zero DB Calls)

The LLM has two tools:

| Tool | Description |
|------|-------------|
| `get_next_segment()` | Returns the next story part from memory and advances the position counter |
| `repeat_current_segment()` | Returns the current part again without advancing (for re-reads) |

The LLM calls `get_next_segment()` at the start (Part 1) and again after each child interaction. When all parts are narrated, the tool returns `[THE END]` and the agent wraps up.

### At Session End

Transcript is extracted from the agent's `ChatContext` and passed to `session_summarizer.summarize_session()` — the same summarization pipeline used by the voice agent. The summary is saved to the `session_summaries` table.

## Running the Story Agent

```bash
# Development (auto-reloads)
python story_entrypoint.py dev

# Production (alongside voice agent + emotion detector)
bash start.sh   # story_entrypoint.py is started automatically
```

## Frontend Integration Checklist

- [ ] Add a genre selector UI (adventure, fable, animal, fantasy, mythology, moral)
- [ ] Call `GET /api/stories/random?genre=<selected>` to get a preview
- [ ] Show story title + "Listen to this story?" confirmation dialog
- [ ] On confirm: call `POST /api/story-token` with the `story_id`
- [ ] Connect to LiveKit room using the returned `token` + `livekit_url`
- [ ] No special RPC handlers needed — Story Mode does not use emotion detection
- [ ] On disconnect: display a "Story Complete!" or "See you next time!" screen

## Testing

Run the integration test suite against a running server:

```bash
python tests/test_story.py
```

The test script automatically creates a throwaway account + child profile, then covers all endpoints including auth edge cases.
